# Default to header based auth if not specified by the FE

<DecisionHeader status="proposed" lastUpdate="2022-11-22" created="2022-10-25" deciders={["rishabhpoddar", "porcellus"]} proposedBy={["porcellus"]} />

## Context and Problem Statement

We have to decide which auth mode to use if it's not specified by the FE during session creation.

## Considered Options

* Header
* Cookie
* Cookie if rid is present, header otherwise

## Decision Outcome

Header

## Pros and Cons of the Options

### Header

<ArgumentPro> Useful for curl, wget, and in situation where we have no SDK (e.g.: SSR, manual request) </ArgumentPro>
<ArgumentCon> Harder to use in Postman </ArgumentCon>
<ArgumentCon> Support for older SDKs? </ArgumentCon>
<ArgumentNeut> If we have an updated SDK present all solutions work equally well </ArgumentNeut>

### Cookie

<ArgumentCon> Cookies can be hard to use without a browser/our SDK  </ArgumentCon>
<ArgumentPro> Supports older SDK versions naturally </ArgumentPro>
<ArgumentNeut> If we have an updated SDK present all solutions work equally well </ArgumentNeut>

### Cookie if rid is present, header otherwise

<ArgumentPro> Useful for curl, wget, etc. </ArgumentPro>
<ArgumentPro> Useful in situation where we have no SDK (e.g.: SSR, manual request) </ArgumentPro>
<ArgumentCon> Harder to use in Postman </ArgumentCon>
<ArgumentCon> Harder to communicate - seemingly unrelated rid header modifies this </ArgumentCon>
<ArgumentPro> Supports older SDKs </ArgumentPro>
<ArgumentNeut> If we have an updated SDK present all solutions work equally well </ArgumentNeut>
