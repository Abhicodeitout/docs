# Add a new verifyAccessToken function that takes the token as a string

<DecisionHeader status="proposed" lastUpdate="2022-12-13" created="2022-12-13" deciders={["rishabhpoddar", "porcellus"]} proposedBy={["porcellus"]} />

## Context and Problem Statement

In some cases, the BE may need to validate access tokens based on the string only, with no access to the request/response objects.

## Considered Options

* Recommend using a jwt verification lib
* Add a new `verifyAccessToken` function that takes the token as a string

## Decision Outcome

We've decided to add a new `verifyAccessToken` function that takes the token as a string:
- This will be very similar to the existing `getSession`/`verifySession`
- It will need to add a new type that makes it explicit that you can't update the session on the FE through it.

### Recommend using a jwt verification lib

<ArgumentPro> No implementation effort </ArgumentPro>
<ArgumentCon> No access to claim validators </ArgumentCon>
<ArgumentCon> Can't check blacklist </ArgumentCon>
<ArgumentCon> Recommends not using our SDK </ArgumentCon>

### Add a new `verifyAccessToken` function that takes the token as a string

<ArgumentPro> Provides access to validators </ArgumentPro>
<ArgumentPro> Can return nicely typed object that disallows modification of the session </ArgumentPro>
<ArgumentCon> A new function in the interface</ArgumentCon>
