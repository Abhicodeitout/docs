# Not support OAuth w/ header-based auth initially

<DecisionHeader status="proposed" lastUpdate="2023-05-11" created="2023-05-11" deciders={["rishabhpoddar", "porcellus"]} proposedBy={["porcellus"]} />

## Context and Problem Statement

Header-based auth requires a separate implementation using frontend redirection.

## Considered Options

* Always use frontend redirection
* Add support for header-based auth using FE redirection
* Not support OAuth w/ header-based auth initially

## Decision Outcome

Chosen option: Not support OAuth w/ header-based auth initially

* Faster first version
* We can easily add this later
* We are OK with not supporting header-based auth (at least initially) since we think that it'll be a very rare usecase

## Pros and Cons of the Options

### Always use frontend redirection

<ArgumentPro> Same implementation for cookie and header based auth </ArgumentPro>
<ArgumentCon> Goes somewhat against the recommendations (some security concerns) </ArgumentCon>
<ArgumentCon> Very slow - even if the session exists and no auth is required, the full react app would have to load before we can do anything </ArgumentCon>

### Add support for header-based auth using FE redirection

<ArgumentPro> Supports all usecases </ArgumentPro>
<ArgumentPro> Faster if using cookie-based auth (which is the default/common auth mode) </ArgumentPro>
<ArgumentCon> More implementation time </ArgumentCon>

### Not support OAuth w/ header-based auth initially

<ArgumentPro> We can get the first version supporting OAuth2 out faster </ArgumentPro>
<ArgumentNeut> We can easily add support later </ArgumentNeut>
<ArgumentCon> Limits some use-cases </ArgumentCon>
